## 🎯 Оценка «Удовлетворительно» (3)

### Задание 1. Проектирование и запросы в реляционной СУБД MySQL (Модуль 2)
Разверните локальную базу данных MySQL и выполните следующие шаги:
1. **Создание таблиц (`CREATE`):** Создайте две таблицы со связью «один ко многим»:
   * `departments` (отделы): `id` (INT, Primary Key, Auto Increment), `name` (VARCHAR).
   * `employees` (сотрудники): `id` (INT, Primary Key, Auto Increment), `first_name` (VARCHAR), `last_name` (VARCHAR), `salary` (DECIMAL), `hire_date` (DATE), `department_id` (INT, Foreign Key к таблице `departments`).
2. **Наполнение данными:** Добавьте не менее 3 записей в таблицу отделов и не менее 7 записей в таблицу сотрудников.
3. **Написание SQL-запросов:** Составьте три отдельных SQL-скрипта:
   * **Фильтрация (`WHERE`):** Выберите всех сотрудников, у которых заработная плата выше определенного значения, а дата найма — после 1 января 2024 года.
   * **Сортировка (`ORDER BY`):** Выведите список всех сотрудников, отсортированный по фамилии в алфавитном порядке, а при совпадении — по убыванию заработной платы.
   * **Объединение таблиц (`JOIN`):** Напишите запрос с использованием `INNER JOIN`, который выводит имя, фамилию сотрудника, его зарплату и название его отдела.

### Задание 2. Аналитический обзор архитектур NoSQL (Модуль 3)
Подготовьте структурированный аналитический отчет (в текстовом виде), описывающий экосистему NoSQL на основе уроков 3.1–3.5:
1. **Сравнительный анализ:** Заполните таблицу сравнения четырех типов NoSQL баз данных:
   * *Документные*
   * *Ключ-значение (Key-Value)*
   * *Колоночные (Column-family)*
   * *Графовые*
2. **Критерии сравнения:** Для каждого типа укажите: основную единицу хранения данных, характерную СУБД-представителя, масштабируемость (горизонтальная/вертикальная) и один целевой бизнес-сценарий (Use Case), где данный тип базы данных превосходит реляционные решения.

## 🚀 Оценка «Хорошо» (4)

### Задание 1. Архитектура Hadoop и логика MapReduce (Модуль 4)
Продемонстрируйте понимание распределенной файловой системы HDFS и парадигмы вычислений MapReduce:
1. **Архитектурный разбор HDFS:** Пошагово опишите алгоритм взаимодействия между компонентами `Client`, `NameNode` и `DataNode` при попытке пользователя записать файл размером 500 МБ в HDFS при стандартном размере блока 128 МБ и факторе репликации 3.
2. **Разработка алгоритма MapReduce:** Напишите на языке Python (или в виде подробного псевдокода) логику распределенной обработки текстового лог-файла веб-сервера для подсчета количества запросов с каждого уникального IP-адреса.
   * Реализуйте функцию `mapper(line)`, которая принимает строку лога, извлекает IP-адрес и генерирует пару `(IP, 1)`.
   * Реализуйте функцию `reducer(ip, counts)`, которая агрегирует все единицы для конкретного IP-адреса и выводит итоговую сумму `(IP, total_count)`.

### Задание 2. Распределенная фильтрация и агрегация в Apache Spark (Модуль 5)
Используя принципы функционального программирования и ленивых вычислений (Lazy Evaluation) в Spark, решите следующую задачу:
1. **Условия:** Дан датасет транзакций интернет-магазина в формате CSV (поля: `transaction_id`, `user_id`, `amount`, `status`, `timestamp`).
2. **Реализация (PySpark):** Напишите код скрипта, который:
   * Инициализирует сессию Spark (`SparkSession`).
   * Читает CSV-файл в распределенный DataFrame.
   * Выполняет трансформацию: фильтрует записи, оставляя только транзакции со статусом `SUCCESS` и суммой (`amount`) строго больше 1000 рублей.
   * Производит параллельное программирование агрегации: группирует данные по идентификатору пользователя (`user_id`) и вычисляет общую сумму затрат каждого пользователя.
   * Сортирует результат по убыванию общей суммы и сохраняет топ-100 строк обратно на диск в формате Parquet.

## 🏆 Оценка «Отлично» (5)

### Задание 1. Проектирование и разработка сквозного ETL/ELT процесса (Модуль 6 и Воркшоп 1)
Разработайте полноценный скрипт на Python для сбора, обработки и сохранения данных:
1. **Extract (Извлечение):** Напишите функцию, которая подключается к любому открытому REST API (например, JSONPlaceholder, OpenWeatherMap или публичные котировки валют) с помощью библиотеки `requests`, извлекает массив данных в формате JSON и проверяет статус ответа (обработка ошибок 4xx/5xx).
2. **Transform (Трансформация):** Реализуйте очистку данных: удалите дубликаты, отфильтруйте пустые или поврежденные значения (null/None), приведите текстовые метки времени к единому стандарту ISO 8601, добавьте новое расчетное поле (например, флаг актуальности или конвертированное значение).
3. **Load (Загрузка):** Настройте подключение к локально или облачно развернутой документно-ориентированной базе данных MongoDB (используя библиотеку `pymongo`). Скрипт должен атомарно загружать очищенные документы в коллекцию `api_data_storage`.

### Задание 2. Создание и автоматизация конвейера в Apache Airflow (Модуль 7)
Интегрируйте разработанный в предыдущем задании ETL-процесс в промышленный инструмент оркестрации:
1. **Разработка DAG:** Создайте файл направленного ациклического графа (`dag_etl_pipeline.py`) для Apache Airflow.
2. **Конфигурация параметров:** Настройте параметры расписания: конвейер должен запускаться автоматически **ежедневно в 10:00 утра**, начиная с фиксированной даты в прошлом, и иметь лимит на количество одновременно выполняемых запусков (`max_active_runs=1`).
3. **Структура тасок (Tasks):** Граф должен состоять минимум из 3 связанных операторов (используйте `PythonOperator`):
   * `task_validate_source` — проверяет доступность внешнего API (отправляет пинг/хед-запрос).
   * `task_execute_etl_process` — вызывает функции извлечения и трансформации данных, передавая промежуточный результат или статус через XCom (или локальное кэширование).
   * `task_load_to_mongodb` — подключается к MongoDB и осуществляет финальную вставку документов.
4. **Управление зависимостями:** Задайте четкую последовательность выполнения задач в коде DAG с использованием оператора битовых сдвигов (`>>`).

### Критерии оценивания:
* **На 3:** Код SQL-запросов синтаксически корректен, таблицы созданы с соблюдением целостности (Foreign Key), отчет по NoSQL содержит точные определения.
* **На 4:** Понимание архитектурных особенностей HDFS продемонстрировано без ошибок в расчете блоков, код PySpark не содержит синтаксических ошибок и использует правильные методы агрегации Spark SQL / DataFrame API.
* **На 5:** Продемонстрирован работающий ETL-скрипт, база данных MongoDB успешно принимает JSON-документы, а файл DAG для Airflow написан без синтаксических нарушений API Airflow и корректно строит граф зависимостей.
